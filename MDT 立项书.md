**项目名称：** 医院智能融合 MDT 会诊平台  
**申请团队：** [七智医助团队]  
**日期：** 2025 年 12 月 12 日  
**版本：** V1.0

---

## 1. 项目背景与目标

### 1.1 项目背景
随着医疗精细化发展，多学科诊疗（MDT）已成为解决疑难杂症的标准模式。然而，传统 MDT 存在专家时间协调难、患者数据汇总繁琐、会诊记录非结构化等痛点。院方目前急需一套独立、高可用的系统，打破科室壁垒，引入人工智能（虚拟专家）辅助决策，实现全院级的高效协作。

### 1.2 项目目标
研发一套**独立部署、高可用、支持虚实专家融合**的 MDT 会诊系统。
*   **全院覆盖：** 支持跨科室医生发起和参与会诊。
*   **虚实融合：** 引入 AI 虚拟专家（基于大模型），在会诊前/中提供辅助诊断建议，与真人专家共同协作。
*   **实时协同：** 支持同一患者会诊时，多名专家终端的数据实时同步、影像同步阅片、文书协同编辑。
*   **数据闭环：** 实现从会诊申请、讨论、报告生成到随访的全流程数字化。

## 2. 核心需求与功能范围

基于“精神心理医疗助手”的技术积累，我们将能力泛化至全科室，核心功能如下：

### 2.1 业务流程模块
1.  **会诊中心（工作台）：** 可视化看板，展示待会诊、进行中、已完成的病例。
2.  **全病程数据视图（Patient 360）：** 自动聚合 HIS/EMR 中的患者数据（病历、检查、检验、影像），以时间轴展示。
3.  **多端实时协同（核心难点）：**
    *   **同步白板/病历：** 采用 WebSocket 技术，支持多位专家同时查看病历，光标位置实时同步，协同编辑会诊意见。
    *   **音视频会议：** 嵌入实时音视频（RTC），支持屏幕共享、影像同步标注。
4.  **虚拟专家（AI Agent）：**
    *   **预问诊/预分析：** 在真人专家介入前，AI 读取病历，生成“病情摘要”和“初步诊疗建议”。
    *   **会议助手：** 实时记录会议语音，转写为文字，并自动生成结构化 MTD 会诊报告草稿。

### 2.2 系统管理模块
1.  **专家库管理：** 维护全院医生权限、排班状态。
2.  **科室/角色权限：** 严格的数据访问权限控制（RBAC）。

### 2.3 非功能性需求（高可用）
*   **并发支持：** 支持全院 X 个科室同时进行会诊。
*   **容灾备份：** 数据库主从热备，服务异常自动熔断与恢复。

### 3.1 核心模块架构
*   **前端层：** Flutter App/Pad、PC Web 端（医生工作站）。
*   **业务中台：** 会诊流转引擎、实时协同服务（WebSocket）、AI 调度中心。

## 4. 研发计划与周期（预估 4 个月）

由于仅有 2 名开发人员，开发全院级高可用系统压力巨大。建议**采用 MVP（最小可行性产品）策略**，分两期走。以下为第一期（核心功能）计划：

*   **第 1-3 周：需求与设计**
    *   PM：调研院方详细需求，输出 PRD 和原型。
    *   Dev：进行技术选型，搭建开发环境，设计数据库。
*   **第 4-10 周：核心功能开发 (Dev)**
    *   完成用户体系、患者 360 视图对接（模拟数据或真实接口）。
    *   **攻坚：** 实现 WebSocket 实时协同编辑功能（这是最耗时的部分）。
    *   接入 AI 接口（复用现有精神心理助手的底层能力，做 Prompt 调整）。
*   **第 11-13 周：联调与优化**
    *   音视频服务集成（建议购买成熟的 RTC PaaS 服务，如声网/腾讯云，避免自研）。
    *   高可用压力测试与 Bug 修复。
*   **第 14-16 周：部署与试运行**
    *   院内服务器部署。
    *   选取 1-2 个重点科室进行试点。

## 5. 资源预算与成本预估

**注意：** 本预算仅包含研发与基础环境成本，不包含产权购买费用。

### 5.1 人力成本
*   **产品经理（2人 x 4个月）：** 负责需求、设计、项目管理。
*   **软件工程师（2人 x 4个月）：** 全栈开发（Flutter前端 + Java 后端）。
*   *风险提示：2 名开发人员极度饱和，建议申请预算外包 1 名前端或测试人员，或申请采购现成的 UI 组件库。*

### 5.2 基础设施与第三方服务（硬性成本）
1.  **服务器资源：** 院方提供私有云服务器（至少 3 台，做高可用集群）及 GPU 服务器（用于本地化 AI 推理，如需）。
2.  **PaaS 服务费：**
    *   **实时音视频 (RTC)：** 按分钟计费，预估初期 5000-10000元/年。
    *   **LLM API (如不本地部署)：** Token 费用。
3.  **开发工具授权：** 如 IDE、设计软件等。

## 6. 风险评估与应对

### 6.1 产权归属风险 (高)
*   **风险：** 目前产权不清，若代码直接部署在医院服务器，未来可能失去控制权或无法转化为 SaaS 产品。
*   **应对：** **在立项合同中明确：**
    1.  本项目的**源代码知识产权归研发方所有**，院方享有永久免费使用权。
    2.  或者，约定为**联合研发**，双方共享专利，但研发方保留商业化推广权利。
    3.  核心算法（如 AI 虚拟专家 Agent）采用 API 形式调用或封装为 Docker 镜像黑盒交付，保护核心资产。

### 6.2 研发资源风险 (高)
*   **风险：** 2 名开发难以支撑“高可用”和“全院级”的复杂并发场景及 HIS 接口对接。
*   **应对：**
    1.  **技术降维：** 初期不追求极致的微服务架构，采用模块化单体架构，降低部署维护难度。
    2.  **外采服务：** 强烈建议音视频、IM 消息通道使用成熟第三方 SDK，不要自研。
    3.  **接口红线：** 明确院方信息科需提供标准数据视图或接口，研发团队不负责清洗 HIS 脏数据。

### 6.3 医疗合规风险
*   **风险：** AI 生成的建议可能存在误差。
*   **应对：** 界面必须显著标注“AI 建议仅供参考，决策由真人专家负责”，并在后台留存所有修改日志。

## 7. 结语
